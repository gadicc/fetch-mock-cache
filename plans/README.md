- **001 first.** It is the verification baseline: every other code plan's
  `pnpm test` gate is network-dependent and tree-mutating until 001 lands.
  "Soft" dependencies can technically be executed before 001, but their test
  runs will need network access and will rewrite `tests/fixtures/http`.
- **005 after 001**: 005 relies on 001's fixture-regeneration mechanism if
  fixtures ever need regenerating, and on the offline harness for its

  headers (deliberate; documented in the plan).

  plan says to skip that item and report.
- **011 after 003**: the cache-modes design builds on `readCache`/`writeCache`
  behaving as documented (promise form currently broken).
- 007, 009, 010, 012 are independent and can run anytime, in parallel.
- **013 after 001, ideally right after 005** (added 2026-07-05 at `a092fef`):

  (different choke points), but both touch `CreateFetchCacheOptions` and the
  same README section, and both are breaking (`feat(security)!`) — ship them
  in one release so users re-record fixtures once.

## Findings considered and rejected

- **Performance category**: nothing to find — test-time tool, no hot paths,
  no N+1-shaped patterns. Not worth auditing again.
- **Memory-store LRU** (`src/stores/memory.ts:13` TODO): caches live for one
  test-process lifetime; eviction solves no observed problem. Not worth doing.
- **Lossy binary-body hashing** (`src/store.ts:99` decodes bytes with a
  non-fatal TextDecoder before hashing — distinct invalid-UTF-8 bodies could
  collide): theoretical; no practical collision scenario in a test-fixture
  tool. Recorded so nobody re-reports it; revisit only if a real collision
  surfaces.
- **`options.id` path traversal in fs store** (`src/stores/fs.ts:61`): the id
  is supplied by the test author about their own machine; README already says
  ids must be valid filenames. By-design for a local dev tool.
- **`fetch(requestObject, requestInit)` init-merging** (init ignored when
  first arg is a Request): real spec deviation but rare usage; deferred —
  noted in plan 002's maintenance notes rather than planned.
- **URL userinfo credentials (`https://user:pass@host/`)**: nothing to
  redact — the WHATWG fetch spec makes `new Request(url)` throw a TypeError
  for credentialed URLs, so they can never reach the cache. (Considered
  during plan 013.)
- **`filenamify-url` major-version upgrade** (pinned exact at 2.1.2): newer
  majors could change generated cache filenames, invalidating every user's
  committed fixtures. Cost outweighs benefit; unflagged by audit. Leave
  pinned.
